What is MCP?

Model Context Protocol (MCP)

MCP is an open protocol that standardizes how AI assistants connect to external data sources and tools. Think of it as a "USB for AI" - a universal way to plug in capabilities.

Why MCP Matters

Before MCP, every AI integration required custom code. MCP provides a standard interface for:

  • Tools - Functions the AI can call (search, database queries, API calls)
  • Resources - Data the AI can read (files, databases, web content)
  • Prompts - Pre-defined prompt templates for specific tasks

Claude Code acts as a host that connects to multiple MCP servers, each providing different capabilities.

Built-in MCP Support

Claude Code comes with several MCP capabilities out of the box:

File System Access

Read, write, and navigate files in your project without manual configuration.

Web Fetch

Retrieve and process content from URLs for research and documentation.

Bash Execution

Run shell commands, manage git, install packages, and run tests.

Code Search

Glob and Grep tools for finding files and searching content.

Checking Available MCP Tools

Use the /mcp command to see what MCP servers are connected:

# In Claude Code
> /mcp

Connected MCP Servers:
- filesystem (built-in)
- Fetch (built-in)
- postgres (custom) - 5 tools available
- slack (custom) - 3 tools available

Using MCP Servers

Configuring MCP Servers

MCP servers are configured in your Claude Code settings. You can add them at project, local, or global level:

// .claude/settings.local.json
{
  "mcpServers": {
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": {
        "DATABASE_URL": "postgresql://user:pass@localhost/mydb"
      }
    },
    "slack": {
      "command": "npx",
      "args": ["-y", "@anthropic-ai/mcp-server-slack"],
      "env": {
        "SLACK_TOKEN": "${SLACK_TOKEN}"
      }
    }
  }
}

Using MCP Tools in Conversation

Once configured, Claude can use MCP tools naturally in conversation:

Example conversation:

User: "Find all users who signed up last week and send a summary to #growth channel"

Claude: I'll query the database and send the summary to Slack.

[Uses postgres.query to get signup data]
[Uses slack.send_message to post summary]

Done! Found 47 new users. Summary posted to #growth.

Environment Variables

MCP servers often need credentials. Use environment variables for security:

# In your shell profile (.zshrc, .bashrc)
export SLACK_TOKEN="xoxb-your-token"
export DATABASE_URL="postgresql://..."
export GITHUB_TOKEN="ghp_..."

# In settings, reference with ${VAR_NAME}
"env": {
  "SLACK_TOKEN": "${SLACK_TOKEN}"
}

Security Warning

Never commit tokens or credentials to git. Use environment variables or a secrets manager. Add .claude/settings.local.json to your .gitignore.

Subagents & Task Tool

What are Subagents?

Subagents are specialized Claude instances that can be spawned to handle specific tasks autonomously. They work in parallel, allowing Claude to tackle complex multi-part problems efficiently.

The Task Tool

Claude Code uses the Task tool to spawn subagents. Each subagent type is optimized for different work:

Explore Agent

Fast codebase exploration. Searches files, finds patterns, answers questions about code structure.

subagent_type: "Explore"
# Uses: Glob, Grep, Read

Bash Agent

Command execution specialist. Git operations, running tests, package management.

subagent_type: "Bash"
# Uses: Bash tool only

Plan Agent

Software architect. Designs implementation plans, identifies files, considers trade-offs.

subagent_type: "Plan"
# Uses: All tools

General-Purpose

Multi-step task handler. Research, code search, complex operations.

subagent_type: "general-purpose"
# Uses: All tools

Spawning Subagents

Claude automatically spawns subagents when appropriate, but you can request it explicitly:

# Example: Ask Claude to use subagents
"Search the codebase for all API endpoints and list them"

# Claude might spawn an Explore agent:
Task tool:
  subagent_type: "Explore"
  prompt: "Find all API endpoint definitions in the codebase.
           Search for patterns like router.get, app.post,
           @app.route, etc. Return a list of endpoints
           with their file locations."

Parallel Subagents

Multiple subagents can run simultaneously for faster results:

# Claude can spawn multiple agents in parallel
User: "Analyze this React app - check component structure,
       find unused imports, and list all API calls"

# Claude spawns 3 agents simultaneously:
Agent 1 (Explore): Search for component definitions
Agent 2 (Explore): Find unused imports with grep
Agent 3 (Explore): Locate all fetch/axios calls

# Results combined into unified response

Background Agents

For long-running tasks, agents can run in the background:

Task tool:
  subagent_type: "general-purpose"
  run_in_background: true
  prompt: "Run the full test suite and report failures"

# You can continue working while the agent runs
# Check status with /tasks command
# Read output file when complete

When to Use Subagents

Good Use Cases

  • - Large codebase exploration
  • - Parallel search operations
  • - Long-running tests
  • - Multi-file analysis
  • - Complex git operations

Avoid When

  • - Simple file reads (use Read directly)
  • - Known file paths (use Glob)
  • - Single command execution
  • - Quick edits to 1-2 files

Model Selection

The --model Flag

Claude Code can use different Claude models for different tasks. Choose based on speed vs capability trade-offs:

# Start Claude Code with a specific model
claude --model claude-sonnet-4-20250514

# Or use model aliases
claude --model opus    # Most capable, slower
claude --model sonnet  # Balanced (default)
claude --model haiku   # Fastest, simpler tasks

Model Comparison

Model Best For Speed Cost
Opus 4 Complex reasoning, architecture, difficult bugs Slower Highest
Sonnet 4 General development, balanced tasks Medium Medium
Haiku 3.5 Quick edits, simple tasks, exploration Fastest Lowest

Subagent Model Selection

When spawning subagents, you can specify which model they should use:

# Use haiku for quick exploration tasks
Task tool:
  subagent_type: "Explore"
  model: "haiku"  # Faster, cheaper for simple searches
  prompt: "Find all files with 'TODO' comments"

# Use opus for complex analysis
Task tool:
  subagent_type: "Plan"
  model: "opus"  # Better reasoning for architecture decisions
  prompt: "Design the authentication system for this app"

Thinking Modes

For complex tasks, you can request extended thinking:

"think more"

Request extended reasoning for medium-complexity problems.

User: "think more about how to
optimize this database query"

"ultrathink"

Maximum reasoning capacity for the most complex challenges.

User: "ultrathink about the
architecture for this system"

Practical Guidelines

1.

Start with Sonnet

Default model handles most tasks well. Switch only when needed.

2.

Use Haiku for Exploration

When spawning multiple search agents, haiku is cost-effective.

3.

Escalate to Opus for Hard Problems

Complex debugging, architecture design, or when Sonnet struggles.

4.

Combine with Thinking Modes

"ultrathink" + Opus for maximum capability on critical decisions.

Pro Tip

You can check which model is being used with /model command. To switch mid-session, use /model sonnet or similar.
